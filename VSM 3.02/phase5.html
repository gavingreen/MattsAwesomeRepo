Task 5.1:
Launch the Beta Test Version of the Site

Primary Role Technical Lead
Supporting Roles DEV, QA
Output Artifacts Successful test results
Completed Production & simulation Testing Checklist (Test Plan)
Supporting Information Test Plan

This Task requires you to setup the site for access by a number of people for the purpose of beta testing. Production-simulation, or beta, testing is designed to simulate the production environment and conditions. 

Solicit feedback about the site from the beta testers.   You might create a short questionnaire or rating form, with a place for their freeform comments, that they complete after they finish testing the site.   To improve the response rate, you could offer them discounts, gifts, or contest entries for prizes when they submit a completed questionnaire.

Ideally, you should solicit a set of beta testers who represent the expected users of the site. These expected users can include:

  • Internal customers
  • External customers
  • Customers using wireless access to the site
  • Customers with physical handicaps
  • Customers using various operating systems, browsers, screen sizes, resolutions, and pointing devices
  • Customers who are native speakers of the target languages to which the site has been translated, for globalized sites
  • Customers in other countries
  • CMA users who will be entering content
  • CMA users who will not be entering content
  • CDA users who will be entering content
  • CDA users who will not be entering content
  • Be sure to include people to review the site that are not part of the development team.   Include CMA users who will and will not be inputting content. Do the same for the CDA.
 

Task 5.2:
Perform Performance Tests

Primary Role Quality Assurance
Supporting Roles TL, DEV
Input Artifact Test Plan
Output Artifacts

Test results

Completed Testing Checklist (Test Plan)
Supporting Information Performance Advanced Service

This Task requires running the performance tests and scripts that were developed for the Test Plan document, analyzing the results, and then trying to improve the performance of the site, if necessary. Most successful sites allocate a minimum of three weeks for performance testing.

Perform Performance Tests

Take the following into consideration when running the performance tests:

  • Execute the load and stress tests according to the Test Plan.   Comprehensive performance testing includes both stress and load testing.   Stress testing is subjecting the component or system to abnormally heavy volumes of activity for a short period of time.   Load testing is subjecting the component or system to heavy, but not abnormally heavy, volumes of activity for an extended period of time.
  • Consider hiring the performance specialists with EMC Advanced Services to assist with performance analysis and tuning.
  • Determine whether client performance testing tools are appropriate. If not, consider 'leased' tools via prearranged agreements with EMC.  (These should have been bought much earlier.)

Evaluate Performance Results and Identify Problem Areas

Supporting Roles TL, DBA
Output Artifacts Acceptable performance results
Supporting Information Performance Advanced Service

Gather the performance test results in the defect tracking log of performance-related problems and template measurement tools printouts.  Gather additional performance information as needed, such as:

  • Speed of the home page.   To get this information:
    • Turn off AUTO-LOAD images.
    • Go to the home page.
    • Hold down the <Shift> key while you click Reload on your browser.
    • Count the number of seconds before the home page redisplays.
  • Percentage of the site that is cached.  To get this information:
    • Select the Template View folder in the Custom WebSite Production Center tools.
    • Select the template filter.
    • Set the filter to live templates only, for the project your site is using.
    • Record the total number of live templates.
    • Do the following query to get the number of cached templates:
      • select count(*) from database template where CACHE='y'
      • Divide the number of cached templates by the total number of templates.
      • Multiple the answer by 100 to get the percentage of the site that is cached.
    • Number of page views delivered per day, hour, minute, or second, depending on what information is available.
    • Maximum number of COMPONENT calls per page.   To get this information:
      • Look in the home page template.
      • Count the number of COMPONENT path calls in the home page template.
    • Total number of template paths.   To get this information:
      • Select the Template View folder in the Custom WebSite Production Center tools.
      • Select the template filter.
      • Set the filter to live templates only, for the project your site is using.
      • Record the total number of live templates.
      • Count the templates that do not have a path. These are the library templates.
      • Subtract the number of library templates from the total number of templates to get the number of template paths.
    • Percentage of CPU usage
    • Site policy for failover

Analyze the performance information to determine:

  • Are any performance problems the result of hardware, or software? Is the database CPU at 100%?
  • Is the caching strategy appropriate? If not, then review the caching design. Is the site being pre-cached by a spider, or is it usage-driven?
  • Are there problems with integrations?
  • Are any problems caused by unexpected types or volumes of usage?
  • Identify symptoms of any performance problems: are they systemic, or related to a few specific pages?
    • Which pages are having problems?
    • Are they pages that are generated by Custom WebSite?
  • Can the problem be remedied by altering the high-level site architecture?
    • Is the network adequate? Is performance where it should be?
    • Is the database adequate?
    • Do we need more CDSs?
  • Are queries a problem?
    • How many queries are executed per page?
    • Are the queries well-constructed?
    • Is the database properly tuned to process the queries efficiently?
  • Review integrations.   Are there any potentially blocking operations?
  • Custom WebSite will wait on response, so ensure that the user is prompted for when actions are necessary for processing to continue.
  • Evaluate number of queries per page.   Are queries well-constructed? Is database tuned to efficiently process queries?
  • Review the data model.   Is the degree of normalization appropriate? There is some tradeoff between normalization, query complexity, and flexibility of the database schema.

Rerun performance testing as necessary to verify the results of performance tuning.

This is a very important step, and includes sharing best practices and knowledge.

Conduct the project debriefing meeting with the key client people, in which you:

  • Give the client a walk-through of the EMC On-line Support System (VOLSS), including how to log an issue with EMC Support, and how to monitor the status of an issue.
  • Verify which client employees will be responsible for the DBA role and other key Custom WebSite roles.
  • Review key information and procedures such as the following with the client personnel to make sure they know them:
    • How to segregate the development environments from the live environments
    • How to migrate software releases from a development environment to a live environment
    • How to return to the previous release, if necessary
  • Solicit lessons learned from meeting participants and document them.
  • If this project is part of a series of incremental development projects, discuss the next steps.
  • Administer the Post-Project Survey to the client personnel attending the meeting, and collect the results.

Then, you need to do the following things besides the project debriefing:

  • Update the project schedule with actual dates.
  • Communicate 'live-site' information to Marketing and Accounting.
  • Review project priorities and define next project.
  • Create a Post-Project Survey, or use the example provided.
  • Administer the Post-Project Survey to the project team members, and collect the results.
  • Hold a meeting to review the results of the Post-Project Survey, and to solicit lessons learned from the project team members, and document them.
  • Complete the Project Transition Checklist.
  • Provide feedback to EMC on how to improve the EMC Solution Methods (EPM2) methodology.  The e-mail alias is EPM2@EMC.com.
