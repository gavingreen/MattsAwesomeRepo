StoryServer 5.0 Browser-Based Workflow

Complete the Workflow section of the Functional Specification. This section describes the workflow used in the CMA, and typically includes a diagram of the workflow accompanied by a description of each workflow task and role.

A workflow consists of tasks or steps arranged in the order they should be carried out to accomplish a particular result.  Often the same set of tasks must be performed on every new record, file, or template.

For these common tasks, you can define project-level default workflows. For each project, you can define up to four default workflows:

  • An underlying default workflow.  This applies to any content type for which you do not define a more specific workflow.
  • A default workflow for records.
  • A default workflow for files.
  • A default workflow for templates.

The content items within the project will inherit workflow from these default project workflows. If all types of content should follow the same workflow, then you need to define only the underlying default workflow.  Or, you can define different default workflows for different types of content, such as records, files, or templates. You can also create a specific workflow for a particular content item.

Identify and define unique workflow sequences with their tasks:

  • Identify the roles that are responsible for each workflow task.
  • Define policies for user roles.
  • Create a prototype of any major workflow change.
  • Coordinate workflow design with phase activities.  
  • Identify groups responsible for each workflow task.
  • Define policies for user roles in VAF.
  • Create a prototype or storyboard if there is a major change in workflow and review it with the end users.

Note: Using Comm Server might affect the workflow solution.

Task 3.11:
Design Custom WebSite Visitor Services

Primary Role Technical Lead
Supporting Role TA
Supporting Information Custom WebSite e-Business Platform Overview: Visitor Services for Custom WebSite

Define login requirements, authorization, and cookie or URL tracking.

Visitor Services for the Custom WebSite e-Business Platform software is a set of functions delivered to you as template commands. These commands let you:

  • Access visitor record information
  • Track the behavior of visitors as they browse your site
  • Profile visitors
  • Personalize the content that visitors see
  • Target campaigns to segments of visitors
  • Deliver content to visitors through push-channel delivery such as e-mail or pager

To use these template commands, you must plan, configure, and administer your Custom WebSite installation, including your storage of visitor information. This guide leads you through this configuration and then describes the template commands in detail.

The Visitor Services commands fall into the following categories:
 
Visitor record commands Get and set attributes for individual visitors or the traits for all the visitors in a segment
Visitor context commands Get and set the information to record from a visitor's behavior while browsing your site, and set how long to cache that information
Profiling commands Create and increment category:keyword counters when visitors browse pages
Personalization commands Customize a visitor's experience of your site based on accumulated profiling data
Campaign commands With the Custom WebSite Relationship Marketing Server, generate events to trigger campaigns and generate the feedback to evaluate the effectiveness of those campaigns
Custom WebSite Communication Server commands Construct an XML delivery document and deliver it to a Communication Server (for "push" campaigns)
Net Perceptions commands Gather and manipulate ratings information from the Net Perceptions Recommendation Engine

For an in-depth exploration of the capabilities and methods within these categories, see the Custom WebSite Visitor Services Guide.

The Visitor Services offer you the ability to:

  • Gather, store, and retrieve information about site visitors and their behavior
  • Personalize the experience of visitors
  • Deliver content as a function of campaigns

Designing and planning are crucial to ensure that you get what you need for your web site.

You assemble many pieces when you set up Visitor Services—keywords, visitor tracking, visitor context management, templates, reports, authorization, visitor storage, and so on. Many factors may affect your design. For example: What do you want to know about the behavior of your visitors? Do you intend to personalize the site? Do you intend to run campaigns? Who is authorized to manipulate Visitor Services data?

An RMS implementation includes the following components:

Task 3.18:
Define Cache Management Strategy (Detailed Design Document Section 12)

Primary Role Technical Lead
Supporting Role TA
Output Artifacts Detailed Design document, Cache Management Strategy section
Supporting Information Detailed Design Document

"Caching and Site Performance"

This Task requires you to fill-in the Cache Management Strategy section of the Detailed Design document.

As you define the cache management strategy for the site, consider the following:  

  • Disk throughput is affected by the number of cached pages, and hardware is affected by the number of non-cached pages.  
  • More cached pages result in less demand on the network, so the policy should be to cache pages as much as possible.
  • Based on benchmarks for StoryServer 4.2.6 on NT and Solaris, one processor can yield about 120,000 page views per hour (this number does not include components) with a 90% cached/10% dynamic site.  Adding one cached component decreases page view yield by about 15-20%.

Designing tasks and considerations:

  • Develop database naming conventions.  Determine whether or not the customer has any predefined database naming conventions, and comply with those if possible.
  • Determine the level of normalization required.
  • Design stored procedures.
  • Design triggers.

Performance Checklist

EMC Advanced Deployment Server

Performance Advanced Service

This Task requires you to complete the first draft of the System Architecture section of the System Architecture document. To do so, you need to define the network topology of the proposed solution and how EMC products will be integrated.

Draw a picture of the logical network layout for the System Architecture document.   Try to identify all of the key components and significant data about them, so the reader of the document can visualize the system will look.

Things to consider while designing the network:

  • Identify the load balancing requirements.  Refer to the performance section of the Requirements Definition document for volume of site traffic that the network must handle.
  • Evaluate the need for routers.
  • Identify the virtual hosting needs.  Custom WebSite supports virtual hosting both by software and by hardware.
  • Identify the required firewall configuration.  The sensitivity of the data will drive the firewall configuration.
  • Design the database configuration and connectivity requirements.
  • Describe the web servers and plug-ins.
  • Identify any necessary redundancy of data, template code, or hardware requirements.   It is good practice to have at least two CAS, CDS, or web servers on the live site to accommodate failover.
  • Design optimal Custom WebSite platform component configuration.  Do not split the CDS from the web server unless you have a security reason to do so.
  • Describe the Content Management Server.
  • Describe the Content Display Server.
  • Describe the Observation Manager.
  • Describe the application servers.
  • Identify all the client browsers that will work on the system, such as Netscape Navigator version 3.0x on all platforms, Lynx, or other browsers.
  • Define the assumed standard configuration for Content Management Application (CMA) client PCs or workstations.  Include all hardware and software that are needed.
  • Specify bandwidth limitations and assumed bottlenecks in the system's server-to-server performance.   Do not try to guarantee performance, but specify the approach necessary to avoid the bottlenecks and prevent performance issues.
  • Define the strategy that allows the desired level of access by clients' client-to-server performance.   This might include techniques such as replication of data, division of processes, or isolation of time-intensive functions.
  • Describe the enterprise-wide staging plan.  This might include separate development and live environments.
  • Describe the specific wireless networks and phone needs and requirements that are not addressed by other Internet technologies.

Performance guidelines to consider:

  • Bandwidth is the most important factor in the performance of a web site.
  • Latency is a close second.
  • Sustained network activity should not exceed 40% of capacity.
  • Peak activity should not exceed 60% of capacity.
  • Multiple address hosting impedes performance.
  • Each web server plug-in may degrade performance by as much as 30%.
  • Content databases should be accessed over the fastest connections available. Database scalability comes from RAM first, processors second.
  • The CDS will yield more pages if it is on the same machine as the web server than it will if they are separated.   All CDSs should point to a central Comm Server.

Make sure the site requirements, including the performance requirements, will be met by the proposed network topology.

The Advanced Deployment Server (ADS) enables you to aggregate managed assets, such as content or code, into logical collections called jobs. After you group the managed assets into a job, you can easily transfer the job to another system. After you move the managed assets to another server, they inherit the workflow of the Custom WebSite project into which you import them. In the event that a job contains application code or content that is later found to be problematic, you can quickly and easily restore the system to its previous state by rolling back the job containing the faulty items.

 

Task 3.29:
Design the System To Meet Availability Requirements, Including Considerations Such as Redundant Systems and Failover Strategies

 If your requirements include high availability and a failover strategy, it is at this point that design decisions regarding those strategies need to be made. Custom WebSite is just another application that uses a database. So, for mirroring, failover, replication, and so on, the industry-standard strategies for fault tolerance apply. What goes on in the database should be of no interest to the mirror, failover, replication, and so on solution. Since we do not expect the flavor of database to change in mid-operation (same libraries, same client path, and so on), these database coordinates are the only dependency that should be required.

If you install and configure Resonate Central Dispatch software, which is shipped on the Custom WebSite e-Business Platform CD-ROM, the Custom WebSite e-Business Platform provides your site with load balancing and failover capabilities.

The Central Dispatch software provides load balancing and failover capabilities for the following Custom WebSite e-Business Platform components:

  • Content Delivery Server (CDS), including the Application Engines (which in turn include the Tcl (ctld) and ASP (asp) Page Generator instances)
  • Observation Management Server (OMS), including the EMC Routers (vrd), Observation Managers (omd), and Campaign Logging Agents (cld)
  • Communication Server (MCS), including the Multi-channel Delivery Agents (mcd) and Delivery Channel Plug-ins (dcp)

Additional information on the actual configuration of Resonate Central Dispatch (sometimes just referred to as Resonate in some of the threaded discussion groups) can be found in the Custom WebSite documentation and by searching global.EMC.com. In addition, the Custom WebSite System Administration & Tuning training covers load balancing and failover.
