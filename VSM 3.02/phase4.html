Conduct Implementation Kickoff

Primary Role Technical Lead
Supporting Information

Functional Specification, Project Standards and Conventions section

Meeting Minutes Form

This Task requires you to conduct an implementation kickoff meeting with all of the project team's members. After the implementation kickoff meeting, team members should be able to read and understand the Functional Specifications, Requirements Definition document, Database Definition document, System Architecture document, and other project documents.   Based on the understanding gained from this meeting and from the specifications in these project documents, team members should be able to construct the proposed application. 

As a result of this meeting, the team should be able to proceed with solution development based on a common understanding of the project; therefore, this meeting should include an overview and discussion of each of the following topics:

  • Known issues
  • Known risks
  • Development environment
  • Project standards and conventions, such as naming conventions, coding style, comments, cookies, and so on.
  • Project logistics
  • Divide and assign the work for the project.
  • Establish coding standards, formatting requirements, and a code review mechanism. 
  • Establish naming conventions for projects, templates, Tcl variables, database tables and fields, cookies, and files, including images, html files, and multimedia files.
  • Design and create the database.  Distribute to all project team members data definitions for tables, cookies, and any site-wide Tcl variables defined in delivery.tcl, so the team can test code against the database.
  • Determine site-wide interface standards, such as link and background colors. Create a site-wide settings template, where variables such as link color can be set and then referenced in site templates.
  • Establish the web site directory structure for docroot.
  • Establish the project folder hierarchy in the Production Center.  Create the high-level folders.
  • Distribute and install template editor macros.

Task 4.3:
Implement Test and Production Environments

Primary Role Technical Architect
Supporting Roles TL, SA
Output Artifacts

Testing environment available for use

Domain names registered with Domain Name Service (DNS)
Supporting Information

Test Plan

System Architecture document

Create the testing and production environments, including web server, database, search engine and browsers, according to the specifications in the Test Plan and the System Architecture document.  The testing environment should be identical to the production environment to avoid unpleasant surprises at the time the site goes into production.

Develop multiple databases or data sets, if necessary, to implement all of the test scripts.   Data preparation is essential for a high-quality testing effort.

Install web server, database, search engine, and browsers as appropriate.

Set up spidering configurations.

Note: A spider is a program that automatically fetches Web pages. Spiders can be used to feed pages to search engines and to test links and relationships within a site. It's called a spider because it crawls over the Web. Another term for these programs is webcrawler. It is much more efficient to automate the testing of links within the site than to test them manually, and spiders can itemize broken or inoperable links in order to ensure that all such errors are corrected.

This Task requires the filling-in of the Test Plan document. The Test Plan should identify quality goals and strategies to which the project will adhere. Complete the Test Plan based on information in the Project Management Plan, Functional Specification, Detailed Design document, Database Definition document, and System Architecture document.

Notes:

  • In smaller engagements this might be an informal process.
  • When possible, the same people should not be assigned to testing and developmental roles.
  • Test plans that incorporate Globalization are much larger and require the use of many new resources.
  • Most successful sites allocate a minimum of three weeks for performance testing.   If the client does not own appropriate automated performance testing tools, purchase or lease them now.
  • Test suites should cover all of the agreed-upon requirements and functions in the Requirements Definition and Functional Specification documents for this project. Map which test scripts will test each requirement and specification.

Develop test scripts that show how to test each function, behavior, and state in each program and page.   Test scripts are required for both manual and automated testing.  Some test scripts for reusable code, tools, or third-party packages might already be available.

Test scripts should not only test the expected functionality, states, or behavior of the site, but should also test all possible functionality, states, or behavior of the site. Ideally, you should create test scripts that cover:

  • Each phase of testing, according to the Test Plan
  • Each function in the Functional Specification
  • Each requirement in the Requirements Definition
  • Each configuration that could occur, according to the Detailed Design document, Database Definition, and System Architecture documents
  • Each interface with other applications or systems
  • Dependencies and prerequisites
  • Usability
  • Reliability

As you develop the test scripts, ensure that the detailed designs are consistent with the high-level specification and requirements.

Develop multiple databases or data sets, if necessary, to implement all of the test scripts.   Data preparation is essential for a high-quality testing effort.

If it is impossible, as it usually is, to fully test each requirement, function, condition, state, or configuration, then you must use the client's goals, success metrics, and acceptance criteria as well as your own experience and common sense to prioritize, analyze, and determine the degree and extent of testing that will be done.

Perform Code Reviews

Primary Role Developer
Supporting Role TL
Output Artifacts Code that is ready to be tested
Supporting Information

Functional Specification,

Project Standards and Conventions section

Database Definition

Code Review Checklist

VAF Developer's Guide, Chapter 17 Module Coding and Reference Guidelines

This Task requires a software developer to review the code that the team is creating for the client's site.

While reviewing the code, try to consider the following:

  • Enforce good Custom WebSite coding practices and guidelines. Review the standards, conventions, and guidelines recommended in the Code Review Checklist, the Template Development Guidelines, and Chapter 17, Module Coding and Reference Guidelines, of the VAF Developer's Guide.  Also, review the project-specific standards and conventions.
  • Verify that the StoryServer Template Development Guidelines were followed, that code is properly commented, and that consistent programming techniques were used.
  • Trace the logic of each template, function, and procedure.
  • Inspect code to identify algorithmic or coding inefficiencies. Identify areas of excessive complexity and see that they are simplified.  The code should be as simple as possible, traceable, efficient, and properly commented.
  • Check for inefficiencies in the generated HTML code. The size of the generated HTML code can be a performance drag.
  • If this is a global site, check for consistent use of I18N API instead of ASCII-specific routines.   Examples are sorting and searching. Check for consistent use of any message catalogs.   Do not use strings in templates!

Test the accuracy, completeness, and adequacy of the Backup and Recovery Plan.

  • Simulate at least one failure of each component in the system, and exercise the appropriate portions of the Backup and Recovery Plan.
  • Simulate at least one major outage, and exercise the appropriate portions of the Backup and Recovery Plan.

Requirements testing is conducted by the end user of a system, and the testing is based on the requirements, to determine if the system works according to specifications and meets the defined requirements.   Validate that the system meets the user's acceptance criteria.   Ideally this should be done as a two-step process:
